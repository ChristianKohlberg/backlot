### 1. [bug] Unconditional socket unlink defeats the daemon singleton election; two daemons can both own state

- **Location:** `src/daemon/index.ts:171` — *security and trust boundaries*
- **Claim:** start() unlinks the socket path before listening, so a second daemon that loses the ping race can delete the winner's live socket and then bind successfully, producing two daemons that both run recover(), both drive the sweeper, and both mutate the journal — exactly what the file's "Singleton guard" comment claims is impossible.
- **Failure scenario:** Two agents run `backlot up` in the same state root within the same ~100ms with no daemon running. CLI-1 and CLI-2 both ping (both fail) and both spawn a daemon, A and B. Interleaving: A pings (false) → B pings (false) → A rmSync (no-op) → A `listen(sock)` succeeds, A sets ownsSocket, writes daemon.pid, starts its 15s sweep timer → B rmSync **deletes A's live socket** → B `listen(sock)` creates a fresh socket, no EADDRINUSE, B sets ownsSocket, overwrites daemon.pid, starts its own sweep timer. Both daemons now hold the same journal.db and both sweep. All clients reach B. B binds an env and marks it `hot` with live dev-servers. A's sweeper — which has an empty `supervisors` map and no knowledge of B's leases — sees that env past `IDLE_TTL`, calls claimForTeardown, runs a no-op stopAll, and writes `state = 'warm'` back to the journal (engine.ts:1014-1029). On B's next `poolGc` (engine.ts:190) that env is no longer in the `live` set, so B `killGroupVerified`s its own running services out from under an active lease: the user's dev-server dies mid-session and `backlot ctx` still advertises its URL. A is also unreachable and unkillable via daemon.pid (which now names B), so it persists until the machine reboots.
- **Evidence:** src/daemon/index.ts:166-173:
```
  if (await pingExisting()) {
    // A healthy daemon already owns the socket — do NOT recover() or listen.
    if (process.send) process.send('ready');
    process.exit(0);
  }
  if (existsSync(sock)) rmSync(sock, { force: true }); // stale socket from a dead daemon

  server.listen(sock, () => {
    ownsSocket = true;
```
The EADDRINUSE concession at :157-163 only fires if the path is still bound at the moment of listen(). Because B unlinks unconditionally first, B's bind finds a free path and succeeds. Both A and B then set `ownsSocket = true` (:174) and run `engine.recover()` (:188). src/cli/client.ts:79-88 makes concurrent daemon starts routine: `ensureDaemon()` pings, and on failure every caller spawns its own daemon, so two CLI/MCP invocations starting together each spawn one.
- **Suggested direction:** Make the bind atomic: bind to a unique temp path in the state dir and `rename()` onto `daemon.sock` (rename is atomic and the loser can detect it lost), or take an exclusive `flock`/`O_EXCL` lock file before any unlink and exit if it cannot be acquired. Only unlink a socket that has been proven dead *while holding that lock*, and re-check ownership (e.g. `fstat` the bound socket vs the path) before setting `ownsSocket`.

### 2. [bug] CLI hangs forever if the daemon dies mid-request; the 15-min RPC timeout is inert

- **Location:** `src/cli/client.ts:65` — *cli/rpc contract and error classification*  · **language-relevant**
- **Claim:** When the daemon's socket is destroyed after the response has started (daemon crash, OOM-kill, SIGKILL, or `backlot daemon stop` racing a concurrent in-flight verb), rpc()'s promise never settles, and `req.setTimeout(15 * 60_000)` does nothing because no 'timeout' listener is attached, so the CLI blocks indefinitely with no exit code at all.
- **Failure scenario:** Agent A runs `backlot up --json` (a 2-minute build streaming progress frames). The daemon is OOM-killed (dev-servers are ~1GB each per the code's own comments), or agent B runs `backlot daemon stop`. Agent A's process never prints anything to stdout and never exits — violating the 'every path yields an exit code 0/1/2/3/64' contract in the worst possible way for an unattended agent, which waits on the subprocess forever.
- **Evidence:** client.ts:61-65: `res.on('end', () => (result ? resolve(result) : reject(new Error('daemon closed the stream without a result frame'))));` / `req.on('error', reject); req.setTimeout(15 * 60_000);` — resolution is wired only to res 'end' and req 'error'. Empirically reproduced on Node v22.23.1 with an exact copy of rpc(): server sends one progress frame then destroys the socket -> neither 'end' nor req 'error' fires; the promise hangs (test printed 'HANG: no resolution after 5s'). A second test confirmed `req.setTimeout(300)` with no listener aborts nothing ('NO-OP CONFIRMED: request still pending 2s after a 300ms timeout'). The clean-close case (res.end() without a result frame) does reject correctly; only abrupt destruction hangs. Daemon side: the RPC 'shutdown' verb (src/daemon/index.ts:84-88) calls `process.exit(0)` 50ms in, destroying any concurrent in-flight response mid-stream — a self-inflicted instance of this trigger.
- **Suggested direction:** Attach handlers for premature close on the response ('close'/'aborted' with result unset, or use res.on('close') checking res.complete) and reject; add a 'timeout' listener that destroys the request with an infra-class error. Classify the resulting failure as infra-error/exit 3.

### 3. [bug] Daemon-unreachable and all client-side crashes exit 2 (env-error) instead of 3 (infra-error)

- **Location:** `src/cli/index.ts:320` — *cli/rpc contract and error classification*
- **Claim:** Every error that reaches main().catch — daemon failed to spawn, socket ECONNREFUSED/ENOENT mid-poll, 'daemon closed the stream without a result frame', spawn ENOENT — is hard-coded to class 'env-error' and exit 2, but decision 0010 defines env-error as 'the environment is at fault; backlot auto-remediates by recycling', which is mechanically wrong guidance when the broker itself is down.
